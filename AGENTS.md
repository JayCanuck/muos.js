# AGENTS.md

Guidance for AI agents and contributors working on this codebase.

## Project Overview

`muos` is a minimal npm library providing TypeScript/JavaScript definitions for [muOS](https://muos.dev/) supported systems. It exports:

- **`System`** — An enum of all supported systems (keyed by PascalCase name, valued by display name).
- **`assign`** / **`defaultAssign()`** — A mapping of common ROM folder names to their `System` values.
- **`readAssignJSON()`** / **`writeAssignJSON()`** — File I/O helpers for muOS assign.json format.
- **`version`** — The muOS version string the data was generated from.

## Architecture

```
src/
  data.ts      ← AUTO-GENERATED. Do not edit manually. Regenerated by scripts/update.ts.
  index.ts     ← Public API. All exports consumed by library users.
scripts/
  update.ts    ← Fetches upstream muOS data via GitHub REST API, generates src/data.ts.
  bump.ts      ← Bumps the patch version in package.json.
tests/
  *.test.ts    ← Unit tests (vitest). Cover data integrity, API behavior, and script logic.
dist/          ← Build output (dual ESM + CJS). Not checked in.
```

## Key Conventions

- **`src/data.ts` is auto-generated.** Never edit it by hand. Run `npm run update` to regenerate.
- **Dual output:** The library ships both ESM (`.js`) and CJS (`.cjs`) via Vite.
- **Minimal dependencies:** Zero runtime dependencies. Dev dependencies are kept minimal; prefer Node.js built-in APIs (Node 22 LTS+).
- **Scripts run via `tsx`:** The `scripts/` directory contains TypeScript files executed with `tsx` (not compiled to JS).
- **Scripts are excluded from npm:** Only `dist/` is published (controlled by `"files"` in package.json).
- **No submodule:** The project previously used a git submodule for the muOS internal repo. It now fetches data from two upstream repos via the GitHub REST API (zero git overhead):
  - **`MustardOS/extra`** — system definitions and assign `.ini` files (Git Trees API + raw file fetches).
  - **`MustardOS/internal`** — `config/system/version` (single raw file fetch).

## Update Pipeline

The weekly update (`.github/workflows/update.yml`) follows this flow:

1. Checkout this repo
2. `npm ci`
3. `npm run update` — fetches the Git tree from `MustardOS/extra` to discover system `.ini` files, fetches their contents in parallel, parses `[friendly]` sections to build assign mappings, fetches `config/system/version` from `MustardOS/internal`, regenerates `src/data.ts`
4. Check if `src/` has changes (`git diff`)
5. If changed: bump patch version → run tests → build → commit & push → publish to npm

## Making Changes

### Adding a new export
1. Add the implementation to `src/index.ts` with JSDoc.
2. Add tests in `tests/`.
3. Run `npm test` and `npm run build` to verify.

### Modifying the update script
1. Edit `scripts/update.ts`. The `CONFIG` block at the top controls the upstream repo URLs, branches, and paths.
2. Exported pure functions (`parseGitHubUrl`, `toEnumKey`, `readSystems`, `parseFriendly`, `resolveAssignments`, `readVersion`, `generateDataFile`) are tested in `tests/update.test.ts`.
3. The `main()` function is guarded and only runs when the script is executed directly.

### Updating upstream source paths
If muOS changes its repository structure, update the `CONFIG` object and `INI_PATH_RE` regex in `scripts/update.ts`.

## Code Style

- TypeScript strict mode enabled.
- Prettier + ESLint enforced (run `npm run fix` to auto-fix).
- Prettier config: 100 char width, 2-space indent, single quotes, no trailing commas.
- Import order: builtins → external → internal → relative (enforced by eslint-plugin-import).

## Testing

- Framework: [Vitest](https://vitest.dev/)
- Run: `npm test` (single run) or `npm run test:watch` (watch mode)
- Pure function tests require no I/O (data passed as arguments).
- All exported functions and script logic should have test coverage.
